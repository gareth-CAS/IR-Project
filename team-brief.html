What this document is

This is the authoritative end-to-end reference for the Image Recognition (IR) rollout across Pick n Pay and Checkers stores. It shows exactly what happens — and what data is needed — at every step from a merchandiser opening the app to compliance data appearing on the dashboard.

Four technology teams are involved: Mac Mobile (field app, server infrastructure, and metadata packaging), Neurolabs (AI image recognition), Keystone / Cognizance (analytics and compliance engine), and Pack & Stack (ranging data). The confirmed data flow is: photo → Mac Mobile → Neurolabs → Mac Mobile (adds metadata) → Keystone → compliance result → Mac Mobile app → merchandiser. Every gap in the data chain is documented here with its owner and status.

How to use this: Use it in every team meeting as the shared reference. When a problem surfaces, find the step it lives in, check the gap, and assign the action. Click any orange gap button to see the full detail.

End-to-end process flow
1
Field — Merchandiser arrives at store
01
Merchandiser logs in to the app
Opens Mac Mobile on their handset at the store
Data needed: Merchandiser credentials (username + password) linked to their account in Mac Mobile. Must be set up in advance by Sean's team. If credentials aren't configured, the merchandiser cannot start.
Mac Mobile
Owner: Sean / Mac Mobile
↓
02
App loads the correct store
Merchandiser selects or is auto-assigned the store they are visiting
Data needed: Store ID and name must exist in Mac Mobile's system, and the store must be linked to the merchandiser's route. If the store isn't configured, the merchandiser cannot proceed.
Mac Mobile
Owner: Sean / Mac Mobile
↓
03
App shows the category form and principals
Merchandiser opens a form for the category (e.g. "Snacking") and sees the list of brand principals to cover (e.g. Nestle, Unilever)
Data needed: Ranging list from Pack & Stack must be loaded into Keystone, which feeds Mac Mobile. The ranging list controls which principals appear in the app for this store. If a principal is missing from the ranging list, it will not appear in the app — and compliance will show as zero — even if those products are physically on shelf.

Critical data chain: Pack & Stack → Gareth → Keystone → Mac Mobile
Mac MobilePack & StackKeystone
Owner: Gareth / Jordan / Camelle
2
Capture — Taking the shelf photo
04
Merchandiser takes a photo of the shelf
Standing in the aisle, they photograph the relevant shelf section for the selected category
Data needed: Good image quality — not blurry, not tilted, no glare. The new GCP app (releasing week of 20 April) will automatically reject bad images before upload. Until then, poor photos are a leading cause of recognition failures. Error rate is currently at 42% — up from 10%.
Mac MobileGCP
Owner: Mac Mobile / Neurolabs
↓
05
Photo is uploaded to Neurolabs
The image travels from the handset through Mac Mobile's servers (Vodacom-hosted) to Neurolabs
Data needed: Stable server connection. Mac Mobile's Vodacom servers must be operational and passing images through correctly. Server outages mean the image never reaches Neurolabs — the merchandiser waits and no result is returned.
Mac MobileNeurolabs
Owner: Sean / Mac Mobile
3
AI Processing — Neurolabs identifies what's on shelf
06
AI scans the image and detects products
Neurolabs' image recognition engine looks at the photo and identifies every product it can see on the shelf
Data needed: The photo from step 05. Neurolabs runs the detection automatically. No additional input data is needed at this point beyond the image itself.
Neurolabs
Owner: Toby / Neurolabs
↓
07
AI matches each product to the catalog
Each detected product is matched to a known SKU (product code) using its barcode and operational category
Data needed: A clean, complete product catalog in Neurolabs — with correct barcodes and correct operational categories for every product. If the catalog has duplicates, missing barcodes, or wrong categories, Neurolabs cannot match the product and it is dropped from the results entirely.

Critical sign-off required: Tina (PnP) and Toby (Neurolabs) must confirm the catalog is clean before this step can be trusted.
Neurolabs
Owner: Toby / Tina / Cameron Dawson
↓
08a
Neurolabs sends recognition results back to Mac Mobile
Results do NOT go directly to Keystone — they return to Mac Mobile first
Data passed: SKU code + confidence score (%) per detected product. This goes back to Mac Mobile, not to Keystone. This is a critical step that was previously undocumented.
NeurolabsMac Mobile
Owner: Toby (Neurolabs) → Christo / Sean (Mac Mobile)
↓
08b
Mac Mobile adds metadata and sends enriched request to Keystone
Mac Mobile packages the Neurolabs results with store and session context, then forwards the combined JSON to Keystone
Metadata added by Mac Mobile: Store ID (customer ID), rep identifier, form/category the photo was taken for, and the principal. This is what allows Keystone to know which store, rep, category and principal the Neurolabs results belong to.

Why this matters: Keystone never receives raw Neurolabs output. If the store ID passed by Mac Mobile is a duplicate or stale ID from the old Pack & Stack system, Keystone may process the request against the wrong ranging list or return no result at all.
Mac MobileKeystone
Owner: Christo / Sean (Mac Mobile) → Jordan (Keystone)

09
Keystone checks what should be on shelf
Keystone looks at the ranging list for this store — the master list of what products should be stocked here
Data needed: A complete, store-level ranging list covering all principals. If the ranging list is partial, Keystone has no baseline for those missing principals. Compliance will show as zero for them — even if the products are physically on shelf.

Hard deadline: 17 April 2026 — all ranging lists must be finalised for Checkers and PnP.

How the team should use this document

In every team meeting: Open this document as the shared screen. When a problem is raised, navigate to the step it belongs to and check the gap. Assign the action to the named owner on the spot.

When a gap is resolved: Notify Gareth so the document can be updated. A gap is only resolved when the data is confirmed clean — not when an action has been taken.

The three things that must happen by 17 April for the pilot to proceed: (1) Ranging lists finalised and loaded into Keystone for all pilot stores. (2) Product catalog cleaned and signed off by Tina and Toby. (3) Jordan's example API response with category-level structure shared with the team for review.

Remember: Every gap in this flow has a direct effect on what the merchandiser sees in the app. A bad ranging list means missing principals. A dirty catalog means missed products. A broken API structure means the compliance screen shows nothing useful. The merchandiser experience is the measure of whether we've solved it.
